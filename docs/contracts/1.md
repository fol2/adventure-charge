


# Star Dash Codex Contract

## 1. Project Goal

We are building a small, beautiful, beginner-friendly browser game called **Star Dash**.

The game should be simple at first:
- The player moves left and right.
- Stars fall from the sky.
- The player collects stars to gain points.
- Rocks fall from the sky.
- If the player touches a rock, the game ends.
- The player can restart.

The goal is not to build a huge game immediately.  
The goal is to build a small working game, then improve it step by step.

---

## 2. Important Rule

Do not try to build the whole final game in one step.

Build in small versions:
1. Basic project setup
2. Player movement
3. Falling stars
4. Score system
5. Falling rocks
6. Game over screen
7. Better visuals
8. Sound, polish, and small fun upgrades

Each step must be easy for a beginner to understand.

---

## 3. User Skill Level

The user is a beginner.

Assume the user does not know much about coding yet.

When writing or changing code:
- Use clear names.
- Keep files short where possible.
- Add simple comments.
- Avoid clever tricks.
- Avoid advanced patterns unless truly needed.
- Explain what changed in plain English after each task.

Do not make the project too complex.

---

## 4. Tech Stack

Use:
- JavaScript
- Phaser
- Vite
- HTML/CSS only when needed
- Browser local storage for high score later

Do not use:
- React
- TypeScript, unless the user asks later
- Backend servers
- Databases
- Login systems
- Multiplayer
- 3D engines
- Complex build tools
- Heavy libraries unless absolutely necessary

Keep the game as a simple web game.

---

## 5. Game Style

The visual style should be:

**Cute 2D space arcade game**

The game should feel:
- Colourful
- Friendly
- Easy to read
- Smooth
- Fun
- Not scary
- Not too busy

Suggested assets:
- Cute spaceship or character
- Golden stars
- Cartoon rocks or meteors
- Soft space background
- Big readable score text
- Simple start and restart buttons

All art should feel like it belongs in the same game.

---

## 6. File Structure

Use a clean file structure like this:

```text
src/
  main.js
  scenes/
    MenuScene.js
    GameScene.js
    GameOverScene.js
  objects/
    Player.js
    FallingItem.js
  data/
    gameSettings.js
  ui/
    scoreText.js

assets/
  images/
  sounds/
````

If a file is not needed yet, do not create it early just to look organised.

Start simple, then split files when the code becomes easier to manage that way.

---

## 7. Coding Rules

When editing code:

1. Make the smallest useful change.
2. Keep the game running after every change.
3. Do not rewrite everything unless there is a very good reason.
4. Do not add features that were not requested.
5. Do not silently change the game idea.
6. Use simple variable and function names.
7. Keep gameplay values in `gameSettings.js` when possible.

Example settings:

```js
export const gameSettings = {
  playerSpeed: 300,
  starPoints: 1,
  startingFallSpeed: 120,
  speedIncreaseEveryPoints: 10
};
```

The user should be able to change numbers easily without hunting through the whole project.

---

## 8. Gameplay Rules

The first playable version should have:

* A player at the bottom of the screen
* Left and right movement
* Stars falling from the top
* Rocks falling from the top
* Collision between player and stars
* Collision between player and rocks
* Score text
* Game over state
* Restart option

Do not add power-ups, shops, levels, skins, bosses, or missions until the basic game works.

---

## 9. Visual Polish Rules

Only improve visuals after the game is playable.

Good polish includes:

* Nice background
* Better sprites
* Button hover effects
* Small sparkle when collecting a star
* Tiny screen shake when hitting a rock
* Smooth movement
* Clear score text
* Simple sound effects

Do not add visual effects that make the game hard to read.

---

## 10. Learning Rules

After each coding task, explain:

1. What files changed
2. What the new code does
3. How the user can test it
4. What the next small step should be

Keep explanations short and beginner-friendly.

Do not give a huge technical lecture unless the user asks.

---

## 11. Testing Rules

After making a change, check that:

* The project still starts
* The game screen loads
* There are no console errors
* The player can move
* The new feature works
* Existing features still work

If something breaks, fix that before adding anything new.

---

## 12. Asset Rules

When using generated images:

* Use transparent backgrounds for sprites when possible.
* Keep the same art style.
* Use simple readable shapes.
* Avoid tiny details that are hard to see.
* Use short, clear asset names.

Good file names:

```text
player.png
star.png
rock.png
background.png
collect.wav
hit.wav
```

Bad file names:

```text
image123_final_REALLYFINAL.png
cute thing.png
newnewstar2.png
```

---

## 13. What Codex Should Not Do

Do not:

* Build a massive game in one answer
* Add unnecessary systems
* Add complicated architecture
* Add multiplayer
* Add accounts
* Add a database
* Add payment systems
* Add advanced physics unless asked
* Replace the whole project without permission
* Make the code hard for a beginner to understand

The priority is:

**Simple first. Fun second. Beautiful third. Bigger later.**
